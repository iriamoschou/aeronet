## 2. Refined Logical Models

### 2.1 Refinements Made Following Task 1 Feedback

The logical models produced in Task 1 have been refined in the following ways for Task 2 implementation:

**Relational Model Refinements:**

1. **UserRoleProfile table added** — Task 1 did not capture the role-specific extended attributes described in scenario Section 6 (e.g. Procurement Officer authorization limits, Quality Inspector certification IDs, Equipment Engineer facility assignment). A dedicated table was added to capture these without embedding sparse fields in the core SystemUser table.

2. **EngineeringArtefact table added** — Task 1 noted that CAD files, engineering drawings, and PDF artefacts should be handled separately from the core relational model. A reference table was added to store controlled metadata and storage identifiers for these unstructured artefacts, without embedding binary content in the database.

3. **CHECK constraints added** — All status fields (PurchaseOrder.status, Shipment.status, DeliveredItem.condition_on_arrival) now enforce valid domain values at the database level, improving data integrity beyond what was shown in Task 1.

4. **Indexes added** — Performance indexes were added on all foreign key columns and high-frequency query fields (order_date, status, shipment timestamps, audit log user and timestamp).

5. **Data types specified** — All fields now carry explicit PostgreSQL data types, completing the logical design specification.

**Document Model Refinements:**

1. **QCReports schema aligned with provided JSON samples** — The result_payload structure now correctly accommodates both Dim_NDT_report.json (nested dimensionalTolerance and nondestructiveTesting objects) and EnvironmentalTest_report.json (environmentalTest object with temperatureRange, humidityExposure). The report_type field distinguishes between these variants, avoiding a sparse schema.

2. **CertificationRecord immutability enforced** — An is_immutable boolean field was added. The /api/certifications/<id>/approve endpoint sets this to True on approval, and subsequent edit attempts return a 409 Conflict response. This implements the scenario requirement that "certifications and inspector signatures should be immutable once finalised."

3. **IoTLog alert_flag indexing** — An index on alert_flag and timestamp was added so that the dashboard IoT alert query retrieves only flagged records efficiently.

### 2.3 Document Logical Model (Refined)

Three MongoDB collections. Each references the relational system through stable identifiers (delivered_item_id, iot_device_id) — referencing rather than embedding to avoid data duplication.

**QCReports collection:**
```json
{
  "_id": "<ObjectId>",
  "report_id": "QC-784512-A1",
  "delivered_item_id": 1,
  "report_type": "dimensional_NDT",
  "inspection_timestamp": "2025-08-28T10:00:00Z",
  "inspector_emp_id": "EMP-002",
  "inspector_name": "James Nakamura",
  "outcome_status": "pass",
  "version_number": 1,
  "result_payload": {
    "visualInspection": "Pass",
    "dimensionalTolerance": {
      "result": "Pass",
      "measurements": [
        {"dimension": "length", "measured": 15.002, "unit": "m"}
      ],
      "deviation": 0.002
    },
    "nondestructiveTesting": {
      "type": "Ultrasonic", "result": "Pass",
      "comments": "No internal defects detected."
    }
  },
  "tolerances": {"length_tolerance": 0.005},
  "comments": "All measurements within specification.",
  "is_immutable": false
}
```

The nested result_payload is the key justification for document storage: a visual inspection report omits dimensionalTolerance and nondestructiveTesting; an environmental test uses a different nested structure. Forcing these into a relational schema would produce sparse tables or require excessive table fragmentation.

**CertificationRecords collection:**
```json
{
  "_id": "<ObjectId>",
  "cert_id": "CERT-2025-001",
  "delivered_item_id": 1,
  "certification_type": "Component_Airworthiness",
  "approval_status": "approved",
  "approval_date": "2025-11-14T16:00:00Z",
  "inspector_details": {"emp_id": "EMP-002", "name": "James Nakamura"},
  "digital_signature_ref": "SIG-JN-002-20251114",
  "material_traceability_ref": "BATCH-AC-2025-OCT-07",
  "certification_notes": "Component meets AS9100 requirements.",
  "supporting_evidence_refs": ["/storage/certs/cert-001.pdf"],
  "is_immutable": true
}
```

Once is_immutable is set to true by the approve endpoint, the API returns 409 Conflict on any subsequent edit attempt, implementing the scenario requirement for immutable approved certifications.

**IoTLogs collection:**
```json
{
  "_id": "<ObjectId>",
  "iot_device_id": "IOT-TC9-GPS-01",
  "context_id": "SHIPMENT-TRK-20260108-003",
  "timestamp": "2026-01-08T14:22:00Z",
  "readings": {
    "temperature": -8.5,
    "vibration": 0.05,
    "pressure": null,
    "gps_location": {"lat": 52.3676, "lon": 4.9041, "description": "Rotterdam Port Area"}
  },
  "alert_flag": true
}
```

The readings object varies by device type — temperature sensors omit gps_location; transit containers include it. Document storage accommodates this without schema redesign.

## 3. API Design and Development

All endpoints use JSON. Authentication uses JWT Bearer tokens. The token is obtained from POST /api/auth/login and sent as `Authorization: Bearer <token>` on all subsequent requests.

| POST | /api/auth/login | User login — returns JWT token | email, password → token, user | None | — |
| GET | /api/auth/me | Current authenticated user info | user_id, role, emp_id | Token | — |

### RBAC Enforcement

Role-based access is enforced at the API layer using Python decorators. Two decorators are implemented:

- `@token_required` — verifies JWT, extracts user payload, rejects unauthenticated requests with 401
- `@role_required(*roles)` — verifies JWT and checks that the user's role is in the allowed list, rejects with 403 if not

Every rejected access attempt is written to the AuditLog with action_type='ACCESS_DENIED' and outcome='failure'.

The Auditor role is explicitly restricted to read endpoints only. Write, create, and approve operations return 403 Forbidden for Auditor tokens.

## 5. Deployment Explanation

**Chosen approach:** Local deployment

The system is deployed locally on a single machine running Flask (port 5000), PostgreSQL (port 5432), with MongoDB on Atlas free tier. The frontend is served directly by Flask as static files, eliminating the need for a separate web server.

This approach was selected for the following reasons:

**Reliability for demonstration.** Local deployment eliminates dependency on network connectivity, external hosting uptime, or free-tier cold start delays during the demo session. The system state is fully under the demonstrator's control.

**Functional completeness.** All system components — database interaction, API endpoints, RBAC enforcement, audit logging, and dashboard visualisations — function identically in local deployment as they would in a cloud deployment. There is no reduction in functionality.

**Assessment brief compatibility.** The brief explicitly states that "locally (e.g. on a student or institutional machine)" is an acceptable deployment mode and that "the deployment approach chosen will not affect marking, provided all functional, security, and integration requirements are met."

**MongoDB on cloud (Atlas).** The document database runs on MongoDB Atlas free tier rather than locally. This demonstrates awareness of cloud-hosted document storage, which is the standard deployment model for MongoDB in production. Atlas also provides the dashboard interface for verifying collection contents during the demo.

**Scalability path.** The Flask application is configured with environment variables (DATABASE_URL, MONGO_URI) that make it straightforward to migrate to a cloud platform such as Railway or Render without code changes, should cloud deployment be required in future.
